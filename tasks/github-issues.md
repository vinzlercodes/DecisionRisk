### 1. Implement live MiroFish handoff facades

Labels: `enhancement`, `ready-for-agent`, `area:backend`, `area:mirofish`, `mvp`, `priority:p0`

Dependencies: existing facade skeletons in `apps/decisionrisk-mirofish/backend/app/decisionrisk/facades/`.

Body:

> *This was generated by AI during triage.*

Implement the live MiroFish handoff contract from the source-of-truth PDF. DecisionRisk product code should use facades instead of scattering product behavior across MiroFish internals.

Required facade responsibilities:

- `MiroFishProjectFacade`: create base project, clone per scenario run, track project/run refs.
- `MiroFishGraphFacade`: build graph/world representation from evidence and risk pack.
- `MiroFishSimulationFacade`: run option x scenario x seed simulations and return structured traces.
- `MiroFishReportFacade`: generate raw report substrate and expose a `mirofish_report.*` artifact.
- `MiroFishArtifactFacade`: normalize MiroFish outputs into DecisionRisk artifact format.

Acceptance criteria:

- Facades expose the handoff calls: `create_base_project`, `build_graph`, `clone_project_for_run`, `run_simulation`, `collect_simulation_trace`, `generate_mirofish_report`, `normalize_mirofish_report_to_claims`, and report artifact write.
- Live facade calls produce structured references that can be written into `run_manifest.json`.
- Raw MiroFish output is treated as report substrate only, not final user-facing output.
- Unit tests cover the facade interface with test doubles or replay fixtures.

Source mapping: PDF sections 8 and 16; `audit.md` remaining work for live MiroFish facade methods.

### 4. Implement automatic Verdict Council pipeline after MiroFish output

Labels: `enhancement`, `ready-for-agent`, `area:verdict-council`, `area:backend`, `mvp`, `priority:p0`

Dependencies: issues 1, 2, and 3.

Body:

> *This was generated by AI during triage.*

Implement the mandatory DecisionRisk verdict evaluation layer. The invariant is: no MiroFish report is user-final until it passes through the DecisionRisk Verdict Council.

Pipeline:

`MiroFish run completes -> MiroFish report + traces + graph + metrics + grounding + ClaimRefs -> VerdictCouncilRunner runs automatically -> verdict.json + council_rounds.json + risk_docket.md -> final user response`

Required services:

- `VerdictCouncilRunner`
- `ReportCritic`
- `ClaimRefAuditor`
- `VerdictGateEngine`
- `RiskDocketGenerator`

Acceptance criteria:

- The live pipeline invokes the council automatically after required artifacts are available.
- `run_manifest.json` includes the MiroFish report artifact in live modes.
- Final response source is `verdict.json` or `risk_docket.md`, never raw MiroFish report text.
- Tests prove a completed MiroFish report without council output is not publishable.

Source mapping: PDF section 11; `audit.md` mandatory DecisionRisk verdict evaluation layer.

### 6. Enforce ReportCritic, ClaimRefAuditor, and VerdictGateEngine gates

Labels: `enhancement`, `ready-for-agent`, `area:verdict-council`, `area:evidence`, `mvp`, `priority:p0`

Dependencies: issue 4.

Body:

> *This was generated by AI during triage.*

Implement the gates that prevent unsupported or overclaimed outputs from becoming final.

Required gates:

- MiroFish report claims become ClaimRefs or are marked unsupported.
- `simulation_metrics.json`, `scenario_runs.json`, `grounding_report.json`, and MiroFish report are loaded before final verdict.

Acceptance criteria:

- Validator fails if live outputs include report claims that are neither ClaimRefs nor unsupported.
- Validator fails if final verdict is derived from unsupported assumptions only.
